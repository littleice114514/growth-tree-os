# Life Vitality Tree / 人生生长树

本文件是 Life Vitality Tree 的主规划与保留池，用来帮助后续开发前判断主线、边界、保留想法和扩展风险。它不是最终冻结方案，后续可以根据真实复盘、数据结构和可用性验证继续调整。

## 1. 模块定位

- 英文名：Life Vitality Tree
- 中文名：人生生长树
- 模块名：Life Vitality Tree Canvas

人生生长树不是普通任务树，不是成就树，也不是财富树。它是一棵用来观察人如何生长、枯萎、修复、结果和重新获得生命力的可视化生命体。

它要表达：

- 一个人是否仍然有生命力；
- 是否还能主动选择；
- 是否还能感受生活；
- 是否还能从失败中恢复；
- 是否还能创造；
- 是否仍与真实的人和世界连接；
- 身体和底层支撑是否还能供给生长；
- 今日行动是否喂养了长期主线。

## 2. 当前已确定路线

- Life Vitality Tree / 人生生长树是新的主视觉方向。
- Life Vitality Tree Canvas 是后续主画布模块建议名。
- Obsidian Graph 不再作为产品主线。
- 四季用于反馈当天人生状态，而不是只按现实季节装饰变化。
- 树干代表人生主线与营养通道，不是简单分类列表。
- 树对象身份具有视角相对性：一个方向在整树视角中可能是小枝，在模块视角中可以成为局部主干。
- 文档阶段先定义语义、边界和决策方法，不启动 3D 开发，不改数据库、IPC 或 Zustand。

### v0.2 补充对齐：Adapter / Mapper

Adapter / Mapper 是一层“翻译层”，它不改变原始数据，只负责把现有数据翻译成 Life Vitality Tree 能渲染的格式。

现在需要它的原因：

- 避免直接改数据库；
- 避免把 Life Tree 和旧数据结构强绑定；
- 后续如果数据结构变化，只改 mapper，不重写 UI；
- 让 Life Tree 可以先跑起来，再逐步接入真实数据。

当前映射规则：

- Review -> Leaf / FallenLeaf：普通复盘映射为 `leaf`；标题或正文包含“失控、拖延、游戏、短视频、没完成、中断、卡住、失败”等关键词时，映射为 `fallen_leaf`。
- Node -> Branch / MinorBranch：现有节点按 `nodeType` 转换为 `major_branch`、`minor_branch`、`leaf`、`flower`、`fruit`、`fallen_leaf` 或 `withered_leaf`。
- TreeSnapshot -> Tree Structure：如果有 `TreeSnapshot`，优先使用其中的 nodes / mainlines 生成主干、大枝、小枝和状态摘要。
- 数据缺失或异常：mapper 使用 unknown + 安全读取函数兼容 `createdAt / created_at / createdTime`、`updatedAt / updated_at / updatedTime / lastUpdatedAt` 等字段；缺失字段会 fallback，不让页面崩溃。
- mock fallback：如果没有有效 review / node / treeSnapshot，继续使用 `lifeVitalityTreeMockData` 作为空数据 fallback、UI 演示和后续视觉优化样本。

后续如果要正式接数据库，需要单独设计 GrowthEvent、AnnualRing、TreeSnapshot 等数据结构，不能直接把 v0.2 的前端 mapper 当作最终 schema。

## 4. 核心隐喻：树如何表达人的生命力

人生生长树关注的不是“做了多少事”，而是人的生命系统是否还在持续供给、恢复、连接和创造。

生命力表达包含：

- 底层支撑：身体、睡眠、情绪、财富安全线、关系支持、价值观和长期习惯；
- 主线延续：长期方向是否稳定、是否被日常行动喂养；
- 行动生长：任务、习惯、学习、项目是否形成叶片和新枝；
- 阶段成果：花和果实是否出现，成果是否能沉淀；
- 失败转化：枯叶、落叶和伤痕是否能通过复盘进入土壤，成为后续养分；
- 长期变化：年轮记录每年主线、成果、伤痕、人物、事件和生命力状态。

语言原则：

> 描述状态，不审判人生。

## 5. 生命力判断模型

当前只保留轻量判断模型，不做评分系统。

观察维度：

- 主动性：今天是否仍有主动选择；
- 感受力：是否还能感受生活和身体状态；
- 恢复力：失败、低能量或暂停后是否能修复；
- 创造力：是否有新想法、新表达、新作品或新结构；
- 连接力：是否仍与真实的人、世界和关系连接；
- 支撑力：身体、财富、关系、习惯和情绪是否还能供给主线；
- 长期性：今日行动是否喂养了长期主线。

暂不做：

- 不给用户贴成功或失败标签；
- 不把树形状态写成绝对价值评价；
- 不把生命力简化成单一分数；
- 不用“畸形”作为对用户的直接评价。

## 6. 树对象语义

### 树根

树根代表无法在外部直接看见、但持续供给人生生长的底层系统。具体字段待后续根据复盘、健康、财富、关系模块进一步确定。

保留方向：

- 身体能量；
- 睡眠；
- 健康；
- 价值观；
- 情绪稳定；
- 长期习惯；
- 财富安全线；
- 关系支持；
- 底层能力；
- 复盘后沉入地下的经验。

树根暂不定死，避免过早绑定数据库字段或评分模型。

### 树干

树干是人生主线的集中体现，也是营养通道。它把底层资源供给大枝和小枝。

远景时只看到一根树干；拉近后可以看到树干内部的筋络。筋络代表不同主线内容，例如：

- 学业主线；
- 项目主线；
- 自媒体主线；
- 财富主线；
- 健康主线；
- 关系主线；
- 自我修复主线。

当前建议：

树干粗细不直接等于主线数量，而应综合表示主线稳定度、长期持续时间和底层供给能力。主线过多可能让树干纹理复杂，但不一定代表更健康。

待讨论问题：

- 主线越多，树干是否应该变粗；
- 主线太多是否会导致树干分裂、纹理复杂、供给压力增加；
- 主干粗细是否只表示稳定度，而不是主线数量；
- 主线数量过多是否会造成结构张力。

### 大枝

大枝来自树干，是人生主线向外的展开。它可以理解为主线落地后的模块。

示例：

- 学习；
- 项目；
- 公众号；
- 财富；
- 健康；
- 关系；
- Game Life；
- 时间负债；
- 自我修复。

### 小枝

小枝是从大枝继续衍生出来的具体方向。

例如“学习 C++”可以是一条阶段性主线或大枝，它下面的小枝可能包括：

- 蓝桥杯；
- 力扣周赛；
- C++ 基础语法；
- 算法模块；
- 数据结构；
- 错题集；
- 比赛经验复盘。

对象身份具有视角相对性：

- 从整棵人生树看，C++ 是学习大枝下的小枝；
- 从学习模块内部看，C++ 可以是该模块的主干；
- 从 C++ 分支内部看，蓝桥杯是分支，刷题是叶子，比赛成果是果实。

### 叶子

叶子代表该主线或分支上的每日行动、任务和习惯。

C++ 分支示例：

- 今日刷 2 题；
- 复习前缀和；
- 整理错题；
- 看一节算法课；
- 参加一次模拟赛。

公众号分支示例：

- 写一篇长文；
- 改一篇文章；
- 排版一篇内容；
- 做标题测试；
- 分析一篇对标文章。

叶子状态后续再优化，当前只保留基础语义。

### 花

花代表阶段性小成果，是正在结果前的中间成果。

示例：

- 完成一组练习；
- 完成一个模块原型；
- 写出一篇还不错的文章；
- 跑通一次流程；
- 修复一个关键 bug；
- 完成一次有效复盘。

### 果实

果实是真正拿到结果的东西，包括可沉淀成果。

示例：

- 比赛奖项；
- 项目版本；
- 一套可复用工作流；
- 一篇爆文；
- 一次收入；
- 一个长期有效的习惯；
- 一份年度复盘；
- 一个能持续使用的模块。

果实不只是外部成绩，也可以是可沉淀的经验。例如 C++ 比赛没有拿奖，但形成了一套可复用的比赛经验，也可以结成果实，只是果实类型不同。

### 枯叶

枯叶暂定为还挂在枝上的衰退状态，可能代表拖延、未完成、能量下降、失控风险。

当前不定死细则，后续需结合任务、复盘和时间负债模块判断。

### 落叶

落叶暂定为已经发生并结束的问题、失败、失控或任务中断。

保留方向：

- 落叶可以通过复盘进入土壤；
- 进入土壤后可能转化为根系养分；
- 落叶不是失败判决，而是已经发生的生命事件材料。

### 土壤

土壤是落叶、复盘、经验和旧失败沉淀后的养分层。

当前只保留为隐喻，不急着建模。后续如果接入复盘和年度 AI 总结，可把有效复盘沉淀为土壤养分。

### 年轮

年轮是年度复盘视角，记录从一开始到现在每一年的变化。

含义：

- 每一年的 AI 复盘总结；
- 每年留下的主线变化；
- 年度成果；
- 年度伤痕；
- 年度重大分叉；
- 年度生命力状态；
- 年度关键人物、事件、决策。

交互建议：

- 不常驻显示年轮；
- 可从顶端视角、树干附近按钮或树干详情卡进入；
- 点击后进入年轮视角；
- 每一圈年轮对应一个年份总结。

年轮是新增衍生视角，不影响原本四层视角。

### 四季

四季是当天状态反馈，不是单纯现实季节装饰。

- 春：有成长、新芽、新开始、恢复生机、主动性回升；
- 夏：高投入、高执行、枝叶茂盛、能量充足；
- 秋：结果、沉淀、复盘、收获、可沉淀成果出现；
- 冬：低能量、修复、整理、暂停、根部积蓄力量。

冬天不等于失败。冬天可以表示修复、保存能量、等待下一轮生长。

### 风

风系统进入保留池，暂不进入主线开发。

可能含义：

- 外部环境变化；
- 人际关系影响；
- 市场或平台变化；
- 情绪波动；
- 突发事件。

## 7. 四层视角与年轮视角

### 第一层：远景视角

看到整棵树。

远处只能看到：

- 一棵树；
- 用户名字；
- 大概状态；
- 树干整体；
- 树冠；
- 季节状态；
- 果实、叶片、落叶的大致比例。

不显示太多细节。

### 第二层：结构视角

采用 Hover 摘要 + 点击详情卡。

可吸收 Obsidian Graph 的交互经验：

- 鼠标滑到某个区域时高亮；
- 显示摘要；
- 点击后显示详情卡；
- 不复刻 Obsidian Graph 关系图谱，只吸收 hover / highlight / detail 交互。

第二层可显示：

- 树干由哪些主线筋络构成；
- 大枝代表什么；
- 树根暂定含义；
- 叶子、花、果实、落叶的基本含义；
- 当前区域的创建时间；
- 最近一次更新时间；
- 该区域衍生出的内容数量；
- 下级分支变化带来的最近更新时间。

### 第三层：模块视角

- 哪些地方结果；
- 哪些地方有枯叶或落叶；

### 第四层：细节视角

细节卡对象包括：

- 小枝；
- 叶子；
- 花；
- 果实；
- 枯叶；
- 落叶；
- 根；
- 年轮条目。

细节卡至少保留字段：

- 标题；
- 类型；
- 所属路径；
- 创建时间；
- 最近更新时间；

入口方式：

- 年度 AI 总结；

## 8. 年龄阶段树形与畸形成长描述

年龄阶段树形是待定但重要的设计方向。

不同年龄阶段的人，树形可以不同：

- 少年期：根系、主干初建；
- 青年期：主干形成，大枝快速分化；
- 大学阶段 / 18-25 岁：主线分叉明显，枝干探索多；
- 25-35 岁：树冠扩张，成果和结构稳定度更重要；
- 35 岁以后：果实、生态、年轮、传承权重更高。

年龄树形不是用来判断人生成功或失败，而是帮助用户观察：当前生命结构是否与其人生阶段、身体状态、资源条件和长期目标之间存在张力。

结构描述示例：

- 过早结果：很早出现果实，但根系弱；
- 过早树冠扩张：方向很多，但主干不稳；
- 过晚分叉：年龄阶段到了，但枝干仍少；
- 主干过细、枝叶过密：行动很多，但主线不清；
- 果实很多、叶片灰暗：成就多，但生命力不足；
- 根系弱、树冠大：外部表现强，但底层支撑不够；
- 枝干粗但无叶：结构稳定，但日常行动不足。

以上都先作为结构描述，不作为价值审判。

| 摄像头手势 | 实验 | 不影响核心表达，工程成本高 | 3D 原型可用后单独实验 | 交互、3D |
| 3D 模型 | 后期 | 当前仍在语义定义阶段 | 本地数据映射和 2D/结构原型稳定后 | Tree Canvas、性能 |

| 数据库接入 | 后期 | 当前不改 SQLite schema | 树对象语义稳定并需要真实持久化时 | DB、IPC、迁移 |
| 网页端接入 | 后期 | 当前产品是本地优先桌面端 | 有同步、账号、隐私方案后 | Web、云同步、账号 |
| 多用户推广 | 后期 | 语言、隐私和自定义风险高 | 单用户本地版本稳定后 | 产品规则、隐私、配置 |

| 与时间负债模块关联 | 保留 | Time Debt 仍在 V1 验收阶段 | 需要表达时间透支、恢复和叶片状态时 | Time Debt、叶子、枯叶 |
| 与 Game Life 关联 | 保留 | Game Life 边界未定 | 需要把游戏化反馈接入生命树时 | Game Life、成果、习惯 |
| 与年度 AI 复盘关联 | 保留 | 年轮数据还未实现 | 年度复盘结构稳定后 | 年轮、复盘、AI 总结 |
| 与关系网络 / 他人节点关联 | 保留 | 高隐私且易复杂化 | 关系模块稳定并有隐私边界后 | 关系、隐私、树根 |
| 与未来智能语音助手关联 | 保留 | 不是当前基础闭环 | 语音助手成为真实输入入口后 | 复盘、AI、交互 |

## 10. 决策方法：Life Tree Decision Gate

每个新想法进入 Life Vitality Tree 前，先过五问。

### 五问决策法

1. 它是否增强“生命力表达”？
   - 是：进入候选；
   - 否：谨慎处理。
2. 它是否影响当前基础闭环？
   - 不影响：可进入保留池或实验；
   - 影响：必须延后或拆分。
3. 它是“主线能力”还是“视觉增强”？
   - 主线能力：优先级更高；
   - 视觉增强：后期再做。
4. 它是否会导致工程大重构？
   - 会：先文档归档，不立刻做；
   - 不会：可进入小步验证。
5. 它是否可以被做成可关闭的实验分支？
   - 可以：进入实验；
   - 不可以：必须更谨慎。

### 分类结果

| 分类 | 含义 | 处理方式 |
| --- | --- | --- |
| Core 主线 | 影响生命树核心表达 | 纳入路线图 |
| Backlog 保留 | 有价值但现在不做 | 写入保留池 |
| Experiment 实验 | 可独立验证，不影响主线 | 建实验分支 |
| Archive 归档 | 当前不再推进 | 记录原因，停止干扰 |

### 判断原则

不要因为一个新想法出现，就马上推翻主线。先判断它是：

- 替换主线；
- 补充主线；
- 视觉增强；
- 数据增强；
- 交互增强；
- 独立实验；
- 暂时噪声。

## 11. 未来扩展风险

### 数据库

未来如果 Life Vitality Tree 接数据库，需要考虑：

- 树对象是否独立建表；
- 是否从 review / node / project / wealth / time_debt 映射；
- GrowthEvent 是否需要作为树生长日志；
- 创建时间和最近更新时间如何统一；
- 年轮是否按年度快照生成；
- 叶子、果实、落叶是否需要独立 ID；
- 数据变更后如何触发树重新计算；
- 旧数据如何迁移；
- 树的视觉状态是否存储，还是每次根据数据计算。

当前建议：不要立刻改数据库。先做类型语义和文档定义。

### 网页端

未来如果做网页端，需要考虑：

- Electron 本地数据如何和 Web 同步；
- 是否需要账号系统；
- 是否需要云端数据库；
- 3D 渲染在浏览器性能是否足够；
- 移动端浏览器是否能跑；
- PC 端和网页端的交互是否一致；
- 离线优先与云同步的冲突；
- 隐私数据如何保护。

当前建议：Life Vitality Tree 第一阶段只按本地优先设计，不急着 Web 化，但保留 Web 化需要的数据边界。

### 多用户

如果以后推广给多人使用，需要考虑：

- 生命力评分是否过度主观；
- 不同年龄阶段是否会导致误判；
- 不能把树形状态写成绝对价值评价；
- 不能让用户觉得自己失败；
- 系统应使用描述性语言，而不是审判性语言；
- 需要允许用户自定义模块；
- 不同用户的人生主线不同，不能预设单一成功路径；
- 数据隐私极其重要；
- 年轮、复盘、关系、财富属于高隐私数据。

### 隐私

Life Vitality Tree 可能汇集复盘、财富、关系、身体、时间负债和年度总结。后续如果进入 Web 或多用户阶段，必须优先设计隐私边界、导出/删除机制和本地优先策略。

## 12. 后续最小开发路线

1. 先冻结 v0 语义：树对象、四层视角、四季状态、年轮入口和保留池。
2. 做轻量类型草案：只定义 TreeObject、TreeViewLevel、TreeSeason、TreeStatus，不接数据库。
3. 做静态样例数据：用 seed 或 mock 表达一棵树，不接真实 DB。
4. 做可关闭实验视图：Life Vitality Tree Canvas 作为独立入口，不替换现有主图。
5. 评估从现有 review / node / wealth / time_debt 映射到树对象的最小路径。
6. 数据口径稳定后，再讨论 3D、R3F/Three.js/Drei、手势或网页端。

## 14. v0.4 生命力状态视觉反馈层

Life Vitality Tree v0.4 让 DailyVitalityCheck 不再只是右侧评分，而是进一步生成 LifeTreeVisualState，驱动树的整体视觉气质、顶部状态摘要、详情卡解释和风险/亮点提示。

新增 LifeTreeVisualState：

- tone：整体视觉气质；
- season：四季反馈；
- trunkState：树干 / 主线状态；
- leafState：叶片 / 今日行动状态；
- rootState：根系 / 底层支撑状态；
- fruitState：果实 / 成果状态；
- fallenLeafState：落叶 / 失败复盘状态；
- summary：顶部状态摘要；
- warnings：风险提示；
- highlights：正向变化。

当前映射规则：

- strong_growth：fresh + spring，提示主动性、创造力和方向一致性较好；
- normal_growth：fresh + spring，提示节奏稳定、继续积累；
- repairing：quiet + winter，提示修复、复盘和整理；

- numb：dim + winter，提示感受力、创造力和主动性不足；

- 不做真实 3D；

## 15. v0.5 叶子 / 花 / 果实 / 落叶规则层

Life Vitality Tree v0.5 在 v0.2 mapper 的基础上细化树对象生成规则，让 review / node 不再只粗略映射为叶子或落叶，而是进入更明确的前端状态分类。

本阶段新增的前端规则：

- 普通行动、学习、整理、执行等语义 -> 叶子；
- 跑通、完成一版、阶段性、初步修复等语义 -> 花；
- 发布、验收、收益、可复用、交付等语义 -> 果实；
- 拖延、没完成、中断、卡住、状态差等语义 -> 枯叶；
- 失控、失败、错误、崩掉、浪费、复发等语义 -> 落叶；
- 复盘、归因、总结、调整、规则等语义 -> 土壤候选；
- 防复发、固定规则、模板、习惯、机制等语义 -> 根系养分候选。

当前实现边界：

- v0.5 仍然只改 renderer 前端 mapper / UI 说明；
- 不新增 `root_nutrient` 节点类型，根系养分暂作为 `soil` 的候选说明；
- 不改 SQLite schema，不新增 IPC，不做 AI 自动判断；
- 关键词规则只是启发式，不是最终语义判断；
- 详情卡会解释对象代表什么，并强调落叶不是失败审判，而是复盘转化入口。

v0.5 的重点是让树对象有更清楚的状态语言：行动可以长成叶子，阶段完成可以开花，成果可以结果，失败和失控可以落叶，复盘后的落叶可以进入土壤，形成规则后再成为根系养分候选。

## 16. Life Curve V0 / 人生曲线

Life Curve / 人生曲线不是算命图，也不是未来 80 年预测图。它是 Life Vitality Tree 的趋势层：人生生长树看结构，人生曲线看趋势。

Life Curve V0 先做每日复盘评分系统：

- 个人能力线：学习、技能、认知、表达、专业能力；
- 项目脉冲线：公众号、软件项目、商业尝试、作品发布；
- AI 协作线：Codex、GPT、Claude、自动化工作流、Agent 沉淀；
- 财富安全线：现金流、安全线、固定支出、投资储备、时间负债；
- 健康生命力线：运动、睡眠、精力、身体恢复、情绪稳定；
- 复合人生曲线：前五条线的加权结果。

当前 V0 权重：

- 个人能力线：25%；
- 项目脉冲线：25%；
- AI 协作线：20%；
- 财富安全线：20%；
- 健康生命力线：10%。

当前实现边界：

- 只从 renderer 的 `recentReviews` 前端派生 `LifeCurveDailyScore[]`；
- 每条分数必须带 evidence、risks 和 nextFocus；
- 无复盘时使用 mock fallback；
- 不新增 `life_curve_daily_scores` 表；
- 不改 SQLite，不新增 IPC；
- 不接 AI 自动评分，不做未来预测图；
- 第一版只显示最近 7 天趋势，后续再考虑 30 天趋势、阶段台阶和持久化。

Life Curve V0 最终回答三个问题：

- 今天结束后，哪条线变强了；
- 哪条线在拖后腿；
- 人生底线有没有比昨天更高。
