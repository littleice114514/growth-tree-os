# Claude 接入说明｜growth-tree-os

> 本文档是 Claude（Anthropic）接入本项目开发的固定说明。每次 Claude 新会话首次接入时，按本文执行。

## 1. 开工前必读（按顺序）

每次接入必须按以下顺序读取，不要跳过，不要全量扫描：

1. `AGENTS.md`（仓库根目录）
2. `docs/project-state/CURRENT_STATUS.md`
3. `docs/project-state/NEXT_ACTION.md`
4. `docs/project-state/LOG_INDEX.md`（只读最近 5 条）
5. 与本轮任务直接相关的协议文件
6. 与本轮任务直接相关的代码文件

## 2. 默认禁止读取

- 完整 `docs/dev-log/**` 历史
- 大量旧交接卡
- 无关业务目录
- 无关 assets / 3D 模型 / 构建产物
- 与本轮任务无关的旧阶段记录

## 3. 分支规则

- Claude 参与开发必须使用 `feature/claude-*` 命名的分支。
- 不直接在 `main`、`develop` 或 Codex 正在使用的业务修复分支上修改代码。
- 创建分支前先确认当前最新分支（通常在 `CURRENT_STATUS.md` 的「当前分支建议」中声明）。

```bash
# 示例：从最新分支新建 Claude 分支
git checkout <最新分支>
git checkout -b feature/claude-<任务名>
```

## 4. 协同规则

- Claude 不得与 Codex 同时修改同一批业务文件。
- 如果 Codex 正在某个分支上修改 `TimeDebtDashboard.tsx`，Claude 不应在另一个分支上同时修改该文件，除非明确分工且不会合并冲突。
- Claude 修改业务代码前必须在最终汇报中声明：
  - 允许修改的文件清单
  - 禁止修改的文件清单
  - 验收方式

## 5. Claude 适合承担的任务类型

| 类型 | 示例 |
|------|------|
| 代码审查 | 定位 bug 根因、检查计时逻辑、审查数据流 |
| 方案分析 | 跨天拆分策略、状态持久化方案对比 |
| 小范围修复 | P0/P1 级别的局部修复，不涉及全局重构 |
| 文档补全 | 接入说明、操作卡、日志索引维护 |
| UI 方案推敲 | 组件结构建议、交互流程梳理（不直接改 UI 代码，除非明确允许） |

## 6. 输出格式

每次任务结束时，按以下结构汇报：

```
## 当前理解
## 当前阶段 / 唯一主线 / 下一步任务
## 允许修改范围 / 禁止修改范围
## 本轮调用内容
## 实施动作
## 修改文件
## 验证结果
## GitHub 保存结果
- 是否已 push：
- branch：
- commit：
## 下一步操作
```

## 7. 第一次接入项目的提示词模板

将以下内容直接粘贴到 Claude 新会话中作为第一条消息：

```
你好，我现在要接入 growth-tree-os 项目开发。

本项目已经建立 AI 工作流底座。请按以下顺序执行：

1. 读取仓库根目录 AGENTS.md。
2. 读取 docs/project-state/CURRENT_STATUS.md。
3. 读取 docs/project-state/NEXT_ACTION.md。
4. 读取 docs/project-state/LOG_INDEX.md 最近 5 条。
5. 不要默认读取完整 docs/dev-log。

   - 当前唯一主线

本轮禁止修改任何文件。
本轮只输出项目接入报告，并说明你后续适合承担哪些任务。
```

## 8. 验收纪律

- 文档类任务优先检查文件存在、关键规则接入和 git diff 范围。
- 业务构建只在业务代码、UI、依赖或运行路径变化时执行。
- 如果未运行业务构建，必须写明原因。
- GitHub push 失败时必须输出失败原因和本地状态，不得假装成功。
