# Flexible Ownership Rules｜弹性分工与冗余规则

## 1. 目的

本文件用于保证 Windows 与 Mac 的双端开发既有默认分工，又不会被固定死。

项目默认分工用于减少冲突，不用于限制未来想法。

当新功能、新模块、新想法出现时，必须先判断任务类型、资源需求、文件边界和冲突风险，再决定由 Windows、Mac 或双方协同完成。

---

## 2. 默认分工

### Windows 默认负责

Windows 优先负责：

- 3D 模型开发
- 3D 资源处理
- Three.js / React Three Fiber / 3D 渲染实验
- 模型压缩、贴图、glTF / glb 资源管线
- skills
- .codex / .claude 工作流
- 脚本
- 工具链
- 构建辅助
- 重资源处理
- 文档协议

### Mac 默认负责

Mac 优先负责：

- UI 页面
- 交互体验
- 前端视觉
- 页面布局
- 轻量功能验收
- 用户实际操作体验
- 组件样式优化
- 桌面端体验观察

---

## 3. 默认分工不是硬限制

默认分工不是永久锁死。

允许以下情况：

- Mac 临时参与 3D 页面展示层调整。
- Mac 调整 3D 模块的 UI 容器、交互按钮、布局面板。
- Windows 临时修复 UI 阻塞 bug。
- Windows 为 3D 模块补充前端接入 demo。
- 双方分别处理同一模块的不同层级，例如 Windows 处理 3D 资源，Mac 处理 3D 页面体验。

禁止以下情况：

- 两端同时修改同一个文件。
- 两端同时修改同一批核心组件。
- 两端同时写同一个日志文件。
- 未声明跨边界原因就修改对方默认负责范围。
- 未从 develop 拉取最新内容就直接在旧分支开发新模块。

---

## 4. 跨边界任务规则

如果一端需要修改另一端默认负责范围内的文件，必须先进行跨边界声明。

跨边界声明必须包含：

1. 为什么需要跨边界
2. 要修改哪些文件
3. 是否存在另一端正在开发这些文件
4. 是否可以拆分成独立文件，避免直接改对方文件
5. 本轮日志写入哪里
6. 是否需要先合并 develop
7. 是否需要暂停另一端相关任务

未完成跨边界声明，不允许直接修改。

---

## 5. 跨边界声明模板

每次跨边界前，在日志中写入：

```md
## 跨边界修改声明

### 1. 跨边界原因
说明为什么本设备需要修改非默认负责范围。

### 2. 涉及文件
列出计划修改的文件。

### 3. 冲突风险
说明是否可能与另一端当前任务冲突。

### 4. 替代方案
说明是否可以通过新增文件、适配层、配置层避免直接修改对方文件。

### 5. 最小修改策略
说明本轮只做哪些最小必要改动。

### 6. 回滚方式
说明如果产生冲突，如何回退。
```

---

## 6. 3D 模块特殊规则

3D 模块默认由 Windows 负责重资源和模型管线，但可以拆成两层：

Windows 负责：

- 3D 模型资源
- glb / glTF 文件
- 模型压缩
- 贴图与材质资源
- 3D 渲染实验
- 资源加载性能测试
- 3D 资源目录结构

Mac 负责：

- 3D 页面 UI
- 3D 控制按钮
- 面板布局
- 状态提示
- 用户交互体验
- 轻量预览验收
- 与现有 UI 风格统一

双端协同方式：

如果开发 3D 生长树：

- Windows 建立 3D 资源与渲染底座。
- Mac 建立 3D 模块的页面入口与交互体验。
- 双方不能同时修改同一个 3D 组件文件。
- 如果需要共同修改，应拆成：
  - 资源层文件
  - 渲染层文件
  - UI 控制层文件
  - 页面容器文件

---

## 7. 模块切换弹性规则

当项目从一个模块切换到另一个模块，例如：

- Time Debt → Wealth
- Wealth → Life Vitality Tree
- Life Vitality Tree → 3D Tree
- UI 优化 → 3D 开发
- 3D 开发 → skills 工作流

必须重新判断：

1. 当前模块主要属于 UI、业务逻辑、3D、脚本、工作流还是文档
2. 默认应由哪个设备负责
3. 是否需要另一端协同
4. 是否需要新建 feature 分支
5. 是否涉及已有未合并分支
6. 是否存在文件冲突风险
7. 日志应该写入 mac-*.md 还是 win-*.md
8. 是否需要 integration-summary.md

---

## 8. 冗余设计原则

为了避免未来被固定死，协议遵守以下原则：

- 默认分工优先，但允许有记录地调整。
- 文件边界优先，但允许有声明地跨边界。
- 新想法出现时，先判断任务类型，再决定设备角色。
- 能拆文件就不共改文件。
- 能新增适配层就不直接改核心层。
- 能在 feature 分支验证，就不直接进 develop。
- 能在日志中说明，就不靠聊天记忆。
- 能用协议文件继承，就不靠单个聊天框记住。

---

## 9. 收工检查

每次涉及弹性分工或跨边界修改时，收工前必须额外检查：

- 是否写了跨边界声明
- 是否修改了对方设备默认负责范围
- 是否触碰另一端正在开发的文件
- 是否可以通过拆分文件降低后续冲突
- 是否需要在 integration-summary.md 中说明
- 是否需要更新 PROJECT_HANDOFF_RULES.md
